Repository technical analysis

DeepSWE is a long-horizon coding-agent benchmark corpus.

The repository is not an application runtime; it is a curated suite of reproducible, air-gapped software engineering tasks. Each task pins an upstream repository commit, supplies a Docker environment, provides an agent-facing prompt, and grades behavior through a patch-based verifier.

1. Why DeepSWE exists

The launch blog frames DeepSWE as a response to benchmark contamination, narrow repo coverage, overspecified prompts, and noisy verifiers.

Contamination-free tasks

Tasks are written from scratch rather than copied from public commits or pull requests. Some are motivated by unresolved issues, but the fixes are novel and are not merged upstream, reducing the chance that models have memorized the answer.

Short, realistic prompts

Prompts are intentionally closer to how developers message agents: behavior-focused, shorter than SWE-Bench Pro prompts, and light on prescriptive interface blocks. Agents must explore the repository instead of executing an overspecified checklist.

Cleaner grading signal

The blog reports an LLM-audited verifier disagreement rate of 1.4% for DeepSWE versus 32% for SWE-Bench Pro, where audited errors split into roughly 8% false positives and 24% false negatives.

2. Repository shape

The repo is intentionally shallow at the top level: documentation plus task directories. Complexity lives inside task capsules.

Canonical task contract

tasks/<task-id>/
  task.toml                 metadata, resource limits, image
  instruction.md            prompt shown to the coding agent
  environment/Dockerfile    fallback build recipe for /app
  tests/test.sh             verifier orchestration wrapper
  tests/test.patch          behavior tests applied at grading time
  solution/solve.sh         applies the reference patch
  solution/solution.patch   held-out reference implementation

Design intent

  • Benchmark tasks are reproducible through pinned upstream commits and prebuilt images.
  • Agent submissions are evaluated by observable behavior, not by matching reference code.
  • Reference solutions are present for review and calibration, but are not used during grading.
  • Pier/Harbor metadata separates agent budget, verifier budget, and environment resources.

4. Execution and grading pipeline

Every task uses the same normalized verifier wrapper, differing mainly by pinned commit and task-specific test patch.

Checkout

Dockerfile clones the upstream repository into /app and checks out the task base commit.

Agent edit

The agent receives instruction.md and modifies the cloned project under the configured budget.

Capture patch

Verifier soft-resets to the base commit and writes the model diff to /logs/artifacts/model.patch.

Apply tests

Files touched by tests/test.patch are reset, then hidden or task-added tests are applied.

Score

test.sh base and test.sh new must both exit zero to write reward 1.

Technical signal

Median verifier patches and reference patches are both around 800 lines, which indicates tasks often require broad implementation plus comprehensive tests. The short prompts force agents to recover intent from codebase context, while the large verifier patches encode nuanced behavioral contracts. The blog sharpens this point: DeepSWE prompts are about half as long as SWE-Bench Pro prompts, yet solutions require 5.5x more code and roughly 2x more agent output tokens.

7. Agent performance signal

The blog's most interesting result is that DeepSWE spreads frontier agents into wider, more developer-recognizable gaps than public benchmarks.

GPT-5.5 leads at 70% while also being the most token-efficient top model in the blog's chart. GPT-5.4 is lower scoring but cheaper per trial. Claude Opus 4.7 is close in accuracy to GPT-5.4, but uses more tokens, takes longer, and costs substantially more.

Behavioral takeaways

  • GPT models are described as implementing the visible contract especially literally; GPT-5.5 has the lowest missed-requirement rate in the qualitative analysis.
  • Claude configurations more often miss mirrored requirements in multi-part prompts, such as implementing sync behavior without the async counterpart.
  • Stronger agents self-test more often on DeepSWE. Claude Opus 4.7 and GPT-5.4 authored new tests in more than 80% of reviewed DeepSWE runs.
  • Output tokens, duration, and cost vary by an order of magnitude, but none strongly predicts pass rate.

8. Engineering implications

DeepSWE evaluates end-to-end coding-agent capabilities: repository navigation, build-system fluency, implementation quality, and test-driven correctness.

Strengths

  • Pinned commits and prebuilt images make results reproducible.
  • Air-gapped execution prevents dependency lookup shortcuts.
  • Base and new test modes preserve existing behavior while testing the requested change.
  • Model patch capture supports trajectory review and post-hoc diagnosis.

Operational risks

  • All tasks depend on availability of base images or Dockerfile rebuilds.
  • Large patches can be expensive to inspect manually during curation.
  • Uniform resource limits simplify comparisons but may underfit unusually heavy upstream projects.
  • Reference solutions are visible in-repo, so benchmark consumers must isolate agents from them.

Best use cases

  • Comparing frontier coding agents on realistic multi-file changes.
  • Studying failure modes through captured diffs and trajectories.
  • Sampling deterministic subsets for fast regression testing.
  • Running single tasks to debug agent environment integration.
